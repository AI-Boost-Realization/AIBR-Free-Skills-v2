# Resume Factory — Multi-Agent Resume Variant Pipeline

## Purpose

Generate a portfolio of tailored resume YAML variants, each optimized for a specific company and role. Uses parallel agents for research and scoring, iterates based on recruiter feedback, and applies a final quality gate before declaring variants ready.

## Usage

```
/resume-factory                                    # Full run — all configured targets
/resume-factory --target google:ml-engineer        # Single variant smoke test
/resume-factory --dry-run                          # Print target matrix, no API calls
/resume-factory --force                            # Regenerate all (ignore manifest status)
/resume-factory --skip-research                    # Use existing briefs, re-generate YAMLs only
```

## Paths (configure to match your setup)

```
BASE_RESUME:   ~/resume-pipeline/data/plain_text_resume.yaml
VARIANTS_DIR:  ~/resume-pipeline/data/variants/
BRIEFS_DIR:    ~/resume-pipeline/data/variants/_briefs/
MANIFEST:      ~/resume-pipeline/data/variants/_manifest.yaml
```

## Variable vs Static Resume Fields

**Generate fresh per variant** (these change per company/role):
- `experience_details[].responsibilities` — rewritten bullets targeting this role
- `experience_details[].skills_acquired` — subset and reordering for relevance
- `projects` — selection of 3-4 most relevant projects, descriptions reframed
- `achievements` — reordered by relevance, descriptions optionally sharpened
- `certifications` — include/exclude based on role signal
- `interests` — reframed to align with company culture

**Copy verbatim from base resume** (never change):
- `personal_information` — name, email, phone, location, linkedin, github
- `education_details` — degrees, institutions, dates
- `languages`
- `legal_authorization`
- `work_preferences`
- `availability`
- `salary_expectations`
- `self_identification`

## Execution Flow

### Phase 0 — Pre-flight

1. Parse args (if any) to determine which targets to run
2. If `--dry-run`: print target matrix table and estimated cost, then STOP
3. Verify BASE_RESUME exists and is valid YAML
4. Create VARIANTS_DIR and BRIEFS_DIR if they don't exist
5. Load or initialize MANIFEST:
   - If MANIFEST exists and `--force` not set: load it, skip targets with `status: passed`
   - If MANIFEST doesn't exist: create it with all targets at `status: pending`
6. Report: "X variants to generate, Y already complete (skipping)"

**Manifest initialization template:**
```yaml
base_resume: ../plain_text_resume.yaml
created: {today}
last_updated: {today}
targets:
  - company: company-a
    role: ai-engineer
    display_role: AI Engineer
    status: pending
    current_version: 0
    best_version: null
    best_score: null
    scores: []
    files: []
```

---

### Phase 1 — Evidence Gathering (once, cached)

Check if `{BRIEFS_DIR}/evidence.md` exists and is < 7 days old.
- If YES and not `--force`: skip this phase, use cached evidence
- If NO: invoke the `resume-iterate` skill to gather cross-domain evidence

```
Invoke skill: resume-iterate
Purpose: gather career evidence from GitHub, LinkedIn, projects, etc.
Output will be printed to conversation — save the key evidence summary to:
  {BRIEFS_DIR}/evidence.md
```

Extract and save to evidence.md:
- Top 5 quantified achievements with metrics
- Top technical skills with evidence source
- Key project outcomes
- Strongest career narrative threads

---

### Phase 2 — Research (parallel Haiku agents)

For each target where brief doesn't exist (or `--force` set):

**Batch into groups of 6** and spawn parallel `resume-researcher` agents:

```
Agent: resume-researcher
Model: haiku
Args:
  COMPANY: {company}
  ROLE: {display_role}
  ROLE_SLUG: {role_slug}
  OUTPUT_PATH: {BRIEFS_DIR}/{company}-{role_slug}.md
```

Each researcher agent should:
1. Describe the company's engineering culture, tech stack, and values (from public sources/general knowledge)
2. Extract key requirements for this role type
3. Identify the 3-5 most important signals the hiring team looks for
4. List keywords for ATS optimization
5. Write a one-paragraph "angle" — the strongest framing for a candidate

Wait for all research agents to complete before proceeding.
Report: "Research complete: {N} briefs generated"

---

### Phase 3 — Generate (parallel Sonnet agents)

For each pending target, spawn a `builder` agent:

```
Agent: builder
Model: sonnet
Prompt (per target):
  Generate a tailored resume YAML variant for {COMPANY} — {ROLE}.

  Read these files:
  1. Base resume: {BASE_RESUME}
  2. Company brief: {BRIEFS_DIR}/{company}-{role_slug}.md
  3. Evidence: {BRIEFS_DIR}/evidence.md

  Rules:
  - Copy ALL static fields verbatim from the base resume (personal_information,
    education_details, languages, legal_authorization, work_preferences,
    availability, salary_expectations, self_identification)
  - Rewrite ONLY these variable fields:
    * experience_details[].responsibilities — 4-6 bullets per role, tailored to target
    * experience_details[].skills_acquired — reordered/filtered for relevance
    * projects — select 3-4 most relevant; reframe descriptions toward target role
    * achievements — reorder by relevance, optionally sharpen language
    * certifications — include/exclude based on role signal value
    * interests — reframe to align with company culture signals
  - All facts and metrics must come from the base resume or evidence brief
  - NEVER fabricate experience, metrics, or qualifications
  - Output must be valid YAML matching the schema of the base resume

  Save output to: {VARIANTS_DIR}/{company}-{role_slug}-v1.yaml
  Update manifest at {MANIFEST}: set status=scoring, current_version=1,
  files=["{company}-{role_slug}-v1.yaml"]
```

Batch into groups of 6 max. Wait for completion.
Report: "Generation complete: {N} variants created"

---

### Phase 4 — Score (parallel Haiku agents)

For each variant just generated, spawn a `resume-scorer` agent:

```
Agent: resume-scorer
Model: haiku
Args:
  VARIANT_PATH: {VARIANTS_DIR}/{company}-{role_slug}-v{N}.yaml
  BRIEF_PATH: {BRIEFS_DIR}/{company}-{role_slug}.md
  COMPANY: {company}
  ROLE: {role_slug}
  VARIANT_VERSION: {N}
  MANIFEST_PATH: {MANIFEST}
```

Each scorer plays a senior recruiter at the target company and scores 1-10 on:
- **Relevance** (1-10): How well does experience match this specific role?
- **Specificity** (1-10): Are bullets specific and quantified, or generic?
- **Narrative** (1-10): Does the resume tell a coherent story for this role?
- **ATS Hygiene** (1-10): No keyword stuffing, no generic openers?
- **Authenticity** (1-10): Does it read like real experience?

Write scores + top priority fix to MANIFEST under the target entry.

After all scorers complete, read the manifest and partition results:
- **Passed** (avg >= 7.0): mark done
- **Needs iteration** (avg < 7.0): queue for Phase 5

Report: "{P} variants passed, {F} need iteration"

---

### Phase 5 — Iterate (max 3 rounds per variant)

For each variant needing iteration:

**Identify lowest-scoring dimensions** from manifest scores.
Spawn a `builder` agent with targeted rewrite instructions:

```
Agent: builder
Model: sonnet
Prompt:
  Improve this resume variant based on recruiter feedback.

  Current variant: {VARIANTS_DIR}/{company}-{role_slug}-v{N}.yaml
  Company brief: {BRIEFS_DIR}/{company}-{role_slug}.md
  Score feedback: {paste specific feedback from manifest}

  Top priority fix: {top_priority_fix from scorer}

  Focus improvements on the lowest-scoring dimensions:
  {list dimensions scoring below 7, with their feedback}

  Rules:
  - Do NOT change static fields
  - Fix only what the feedback targets — minimal diff
  - All facts and metrics must still come from base resume/evidence
  - Increment version: save as v{N+1}.yaml

  Save to: {VARIANTS_DIR}/{company}-{role_slug}-v{N+1}.yaml
  Update manifest: current_version={N+1}, add file to files list
```

Then re-score with `resume-scorer`. Update manifest.

Repeat up to 3 total rounds per variant. After round 3, keep the best-scoring version regardless.

---

### Phase 6 — Final Quality Gate (single Opus call)

After all variants have passed or exhausted iterations:

**Compile all passing variant paths** from manifest.

Make a single call using the main conversation context (not a subagent — this needs Opus reasoning):

```
Read all passing variant YAML files.
Check for:
1. Narrative consistency — Does each variant tell a coherent story? No contradictions with base facts.
2. Cross-variant uniqueness — Are the company-specific angles genuinely different, or is it copy-paste with company names swapped?
3. Authenticity — Do the rewritten bullets sound like real experience, not LLM padding?
4. ATS hygiene — No keyword stuffing, no overly generic bullet openers ("Responsible for...")

For each variant that fails a check: note the specific issue.
For variants that pass: mark status=final in manifest.
```

Update manifest with final statuses.

## Error Handling

- If a research agent fails: log to manifest as `status: research_failed`, skip to next target
- If a generation agent produces invalid YAML: retry once with explicit schema reminder, then mark `status: generation_failed`
- If all 3 iteration rounds fail to reach 7.0: keep best version, mark `status: best_effort`, note in final report
- If Opus final gate is rate-limited: skip it, mark all passing variants as `status: passed` (not `final`)

## Non-Negotiable Rules

1. NEVER fabricate experience, metrics, projects, or qualifications
2. NEVER change static fields (personal info, education, legal auth)
3. NEVER send emails, messages, or post to external services
4. All evidence must trace back to base resume or evidence brief
5. Version every file — never overwrite, always increment
